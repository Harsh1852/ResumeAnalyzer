Abstract

This report presents the design, implementation, and evaluation of Resume Analyzer, a cloud-native career intelligence platform built entirely on Amazon Web Services. The system accepts resume uploads (PDF or image), extracts text via AWS Textract and pypdf, and invokes Amazon Bedrock's Claude 3 Haiku model to produce a scored, structured analysis covering role fit, skill gaps, and job-search strategies enriched with real-time web data from the Tavily API. Post-analysis features include live job-board search via Adzuna, AI-generated tailored resumes in Markdown or LaTeX, and a Kanban-style application tracker. The system is composed of eight independent microservices orchestrated entirely through event-driven messaging (Amazon SQS + SNS), deployed via AWS CDK v2 Python, and served by a React 18 single-page application distributed through Amazon CloudFront. This report documents the full architecture, data design, security model, and key engineering trade-offs.

1. Introduction

Modern job seekers face an increasingly competitive market in which resume quality and targeting precision have an outsized impact on outcomes. Generic resume feedback tools either lack current market awareness or require expensive human coaching. Large language models offer an opportunity to deliver expert-caliber, context-aware feedback at near-zero marginal cost.

This project builds Resume Analyzer: a fully serverless, event-driven platform that accepts a user's resume, extracts its text, enriches analysis with live labour-market signals, and delivers a scored report with actionable recommendations, live job listings, and AI-tailored application materials — all without a single always-on server.

The platform is designed around three guiding principles:

  • Decoupled microservices — each service owns exactly one concern and communicates only through queues or topics.
  • Infrastructure as Code — every AWS resource is defined in Python via AWS CDK v2, enabling reproducible one-command deployments.
  • Operational resilience — idempotent handlers, dead-letter queues, and per-service IAM least-privilege contain the blast radius of any failure.

The remainder of this report is structured as follows: Section 2 presents the overall system architecture; Sections 3–8 detail each microservice; Sections 9–12 address cross-cutting concerns (data, messaging, security, CI/CD); Section 13 discusses key design trade-offs; and Section 14 concludes.

2. System Architecture

2.1 High-Level Design

The system adopts an event-driven microservices pattern. Services never call each other directly; instead, they publish messages to managed queues or topics. This ensures that no single service failure can propagate synchronously to another service, and that each service can be scaled, deployed, and tested independently.

3.1 Authentication Service

Authentication is delegated entirely to Amazon Cognito — passwords are never stored or seen by application code. The Auth Lambda acts as a thin adapter between the REST API and the Cognito SDK, translating HTTP requests into Cognito API calls.

On registration, Cognito automatically sends a 6-digit OTP to the user's email address. The Lambda creates a corresponding record in the DynamoDB users table. On verification, the Cognito user is confirmed and the DynamoDB status transitions from PENDING to VERIFIED.

Successful login returns three tokens: an ID token (JWT, 1-hour validity, contains sub/email/name claims), an access token (1 hour), and a refresh token (30-day validity). Downstream API Gateways use a Cognito authoriser to validate the ID token's signature against Cognito's JWKS endpoint, requiring no custom authorisation logic in individual Lambdas.

Account deletion triggers a cross-service cascade implemented via broad IAM permissions granted selectively to the Auth Lambda: it queries and deletes records across all DynamoDB tables and S3 prefixes belonging to that user before calling Cognito DeleteUser.

3.2 Upload Service

File uploads follow the presigned URL pattern — a key cloud-native design choice. Rather than routing file bytes through a Lambda (which has a 6 MB HTTP body limit and charges for upload-time execution), the Upload Lambda generates an S3 presigned PUT URL with a 5-minute expiry. The browser uploads the file directly to S3 at up to 5 GB, then calls POST /uploads/confirm to notify the backend.

The confirm handler records the upload in DynamoDB with status UPLOADED and publishes a parse-trigger message to the SQS ParseQueue. The upload record then tracks state transitions through the pipeline: PENDING → UPLOADED → PARSING → ANALYZING → COMPLETE / FAILED.

3.3 Parser Service

The Parser Lambda is triggered by SQS messages from the ParseQueue with a batch size of 1 and a 360-second visibility timeout (allowing up to 6 minutes for large documents before a redelivery).

Two extraction paths are supported:

  • PDF files — pypdf.PdfReader iterates pages and concatenates extracted text.
  • Image files (PNG, JPG, JPEG) — textract.detect_document_text() performs synchronous OCR, with LINE blocks concatenated in order.

Before processing, the handler checks for an existing parse-job record on the uploadId-index GSI (idempotency guard against SQS at-least-once redelivery). If fewer than 50 characters are extracted, the upload is marked FAILED. Otherwise, extracted text is written as JSON to the parsed-text S3 bucket and a message is sent to the AnalysisQueue.

3.4 Analyzer Service

The Analyzer Lambda is the most computationally intensive service. It runs with 512 MB of memory and a 240-second timeout, reflecting the latency of two sequential external calls (Tavily + Bedrock).

Step 1 — Field Detection

The handler scans the resume text for keyword patterns mapped to professional fields (software engineering, data science, DevOps, finance, marketing, etc.). The detected field drives the Tavily search queries, ensuring market enrichment is domain-specific.

Step 2 — Tavily Market Enrichment

Two concurrent web searches fetch live context: (1) companies actively hiring in the detected field in 2025, and (2) the most in-demand skills for that field. Snippets are concatenated into a market_context string injected into the Bedrock prompt.

Step 3 — Bedrock Invocation

Claude 3 Haiku (anthropic.claude-3-haiku-20240307-v1:0) is invoked via bedrock-runtime.invoke_model with:

  • max_tokens: 8192 — allows a rich, multi-section JSON response
  • temperature: 0.55 — deterministic enough for consistent scoring, creative enough for varied strategies
  • Resume text truncated to 12,000 characters to respect the effective context budget

The prompt instructs the model to act as a senior career advisor with 20+ years of experience and return only a valid JSON object matching a strict schema: 

{
  "resume_score": 73,
  "summary": "...",
  "resume_sections_review": {
    "professional_summary": "...",
    "work_experience": "...",
    "skills_section": "...",
    "education": "...",
    "overall_presentation": "..."
  },
  "critical_improvements": ["...", "..."],
  "top_roles": [
    {
      "title": "Senior Backend Engineer",
      "match_percentage": 82,
      "reason": "...",
      "resume_gaps": ["..."],
      "application_tips": ["..."],
      "target_companies": ["Stripe", "Twilio", "..."]
    }
  ],
  "job_search_strategies": ["..."],
  "skills_to_highlight": ["..."],
  "skills_to_develop": ["..."],
  "key_achievements": ["..."]
}

Score calibration rubric is embedded in the prompt (85–100 = exceptional, 70–84 = strong, 55–69 = average, 40–54 = below average, <40 = needs rewrite), preventing score inflation common in unconstrained LLM outputs.

Step 4 — SNS Fan-out

After writing the analysis JSON to S3, the Analyzer publishes a single SNS message to resume-analyzer-results-topic. SNS delivers the message simultaneously to two SQS subscribers: the ResultsQueue and the NotificationQueue.

3.5 Results Service

The Results Service contains two Lambda entry points in a single handler file:

  • worker_handler — SQS-triggered; reads the analysis from the SNS-wrapped SQS message, writes the full result to DynamoDB (with an idempotency check on uploadId-index), and updates the upload status to COMPLETE.
  • api_handler — HTTP-triggered; serves GET /results, GET /results/{resultId}, and DELETE /results/{resultId} with per-user ownership enforcement.

3.6 Notification Service

Triggered by the NotificationQueue, the Notification Lambda retrieves the user's email address from Cognito via admin_get_user, waits up to 15 seconds (5 × 3 s polls) for the Results Worker to write the resultId to DynamoDB, then sends a formatted HTML email via SES containing the resume score, top matched role, and a deep link to the report.

Importantly, the Notification Lambda does not re-raise exceptions. A failed email is non-fatal — the pipeline result is unaffected, and the user can still access their report through the frontend.

3.7 Jobs Service

The Jobs Service provides three distinct capabilities through a single Lambda with an API handler entry point:

Live Job Search (Adzuna)

When a user requests job listings, the handler calls the Adzuna REST API for each of the candidate's top 5 roles, passing both the role title (what) and the candidate's top skills (what_or). The what_or parameter filters results to postings mentioning at least one of the candidate's skills, producing personalised listings. Results are cached in DynamoDB after the first fetch to avoid repeated API calls.

Course Recommendations (Tavily)

For any job listing, the handler queries Tavily for online courses relevant to the job's title and skill requirements. Course metadata (title, URL, snippet) is cached on the job record in DynamoDB.

Tailored Resume Generation (Bedrock)

The user may request a resume tailored for a specific job in either Markdown or LaTeX format. The handler constructs a prompt containing the original parsed resume text, the job's title and description, and (for LaTeX) a standard academic resume template (Jake's Resume template). Claude 3 Haiku rewrites the resume, preserving structure while optimising content for the target role. Generated resumes are stored in DynamoDB and editable in the browser via a Markdown editor or a CodeMirror LaTeX editor.

3.8 Applications Service

A pure CRUD service with no external API or AI dependency. Each application record tracks company, role, location, source, notes, next action, a complete status history list, and a list of interview rounds. Status transitions drive the Kanban board view. An aggregate GET /stats endpoint computes response rate and offer rate across all applications.

4. Data Architecture

All persistent state is stored in Amazon DynamoDB using on-demand (pay-per-request) billing. DynamoDB was chosen over relational alternatives (Amazon RDS) to avoid VPC complexity, connection-pool management, and idle capacity costs — critical for a student project with bursty, low-volume traffic.

Every access pattern is served by either a table's primary key or a pre-planned Global Secondary Index (GSI). The eight tables and their key design are:

TablePKSKGSIsPurpose
resume-analyzer-users userId – email-index User profile & verification status
resume-analyzer-uploads userId uploadId uploadId-index Upload records & pipeline status
resume-analyzer-parse-jobs parseJobId – uploadId-index Idempotency guard for Parser Lambda
resume-analyzer-analysis-jobs analysisJobId – uploadId-index Idempotency guard for Analyzer Lambda
resume-analyzer-results userId resultId uploadId-index, resultId-index Full analysis output including original resume text
resume-analyzer-jobs jobId – resultId-index, userId-index Cached Adzuna job listings + Tavily courses
resume-analyzer-tailored-resumes resumeId – userId-index, jobId-index AI-generated tailored resumes (Markdown + LaTeX)
resume-analyzer-applications applicationId – userId-index, userId-status-index Job application tracker with interview rounds
Design note: The resume-analyzer-results table stores the original parsed resume text alongside the analysis output. This avoids a cross-service S3 read in the Jobs Lambda when generating tailored resumes. The trade-off is DynamoDB's 400 KB item limit — acceptable in practice since most resumes produce fewer than 15 KB of text.

The composite primary key on the uploads table (userId + uploadId) enforces row-level ownership at the storage layer: a query with a different userId cannot access another user's records, even if the uploadId is known.

5. Messaging & Resilience

5.1 Queue Design

QueueProducerConsumerVisibility TimeoutDLQ Retention
parse-queueUpload LambdaParser Lambda360 s14 days
analysis-queueParser LambdaAnalyzer Lambda300 s14 days
results-queueSNS TopicResults Worker120 s14 days
notification-queueSNS TopicNotification Lambda60 s14 days

Each queue has a corresponding dead-letter queue (DLQ) configured with maxReceiveCount=3. After three consecutive failures, the message is moved to the DLQ and retained for 14 days for manual inspection and replay.

5.2 Idempotency

SQS guarantees at-least-once delivery — a message may be delivered more than once in the event of network issues or Lambda timeouts. Every processing Lambda guards against duplicate processing by querying the relevant GSI for an existing job record before beginning work: 

# Parser Lambda — idempotency guard
existing = parse_jobs_table.query(
    IndexName="uploadId-index",
    KeyConditionExpression=Key("uploadId").eq(upload_id)
)
if any(i["status"] in ("PROCESSING", "COMPLETE") for i in existing["Items"]):
    return  # already processed — safe to discard this redelivery

5.3 Failure Propagation

When a processing Lambda encounters an unrecoverable error, it updates the upload record's status to FAILED and re-raises the exception (causing SQS to retry, and eventually route to the DLQ). The frontend's polling loop surfaces the FAILED status to the user. Notification failures are intentionally swallowed to avoid email spam on retries.

6.2 API Layer

All HTTP calls are centralised in src/services/api.js using Axios. Every authenticated request attaches the Cognito ID token from localStorage as a Authorization: Bearer header. The five API base URLs are injected at build time via Vite environment variables (VITE_AUTH_API_URL, VITE_APP_API_URL, VITE_RESULTS_API_URL, VITE_JOBS_API_URL, VITE_APPLICATIONS_API_URL).

6.3 Key Frontend Features

Status Polling

After upload confirmation, the dashboard polls GET /uploads/{uploadId} every 3 seconds using a setInterval inside a useEffect. When the status transitions to COMPLETE, the interval is cleared and React Router navigates to /results/{resultId}.

Client-Side PDF Generation

The "Download PDF" button uses @react-pdf/renderer to generate a PDF entirely in the browser — no server round-trip. The ReportPDF component defines the document layout using react-pdf primitives, and the output is converted to a Blob and triggered as a file download.

Kanban Drag-and-Drop

The application tracker uses @dnd-kit/core for drag-and-drop. Dragging a card between columns triggers a PATCH /applications/{id} call with the new status, updating both the DynamoDB record and the visual position of the card.

Resume Editor

AI-generated tailored resumes are editable in-browser. Markdown format renders in @uiw/react-md-editor (live split-pane preview); LaTeX format renders in @uiw/react-codemirror with stex syntax highlighting. Edits are saved via PUT /tailored-resumes/{resumeId}.

7. Security Model

7.1 Authentication & Authorisation

All protected API Gateway routes use a Cognito User Pool authoriser, which validates the JWT signature against Cognito's published JWKS before passing claims to the Lambda. No custom token-validation code exists in application Lambdas — the authoriser layer handles it.

Every Lambda that touches user data performs an explicit ownership check: 

user_id = event["requestContext"]["authorizer"]["claims"]["sub"]
item = table.get_item(Key={"uploadId": upload_id})["Item"]
if item["userId"] != user_id:
    return respond(403, {"error": "Forbidden"})

7.2 IAM Least Privilege

Each Lambda's IAM role grants only the permissions it needs. For example:

  • The Parser Lambda can read from the resumes bucket, read/write the parsed bucket, send to the AnalysisQueue, and call textract:DetectDocumentText. It cannot access DynamoDB results, call Bedrock, or publish to SNS.
  • The Analyzer Lambda can invoke only the specific Bedrock model ARN it uses, not all Bedrock models.
  • The Applications Lambda has read/write only on the applications table — no access to any other table or S3 bucket.

7.3 S3 Security

All S3 buckets have BlockPublicAccess.BLOCK_ALL enforced at the bucket level. The frontend bucket is accessible exclusively through CloudFront's Origin Access Control (OAC). Resume buckets are accessible only via presigned URLs (5-minute expiry) or Lambda execution roles.

7.4 CORS

API Gateways are configured with allow_origins=Cors.ALL_ORIGINS to accommodate local development and the CloudFront domain. In a production deployment, this would be restricted to the specific CloudFront domain.

8. CI/CD Pipeline

A GitHub Actions workflow (.github/workflows/deploy-frontend.yml) automates frontend deployment on every push to the main branch where files under frontend/ change. The workflow:

  1. Checks out the repository and configures Node.js 20 with npm caching
  2. Runs npm ci (deterministic install from package-lock.json) and npm run build — Vite injects the API URLs from GitHub Secrets as environment variables at compile time
  3. Configures AWS credentials from GitHub Secrets (AWS_ACCESS_KEY_ID, AWS_SECRET_ACCESS_KEY)
  4. Syncs the dist/ directory to the frontend S3 bucket with differential cache headers: max-age=31536000 for hashed assets (immutable), max-age=0 for index.html (always fresh)
  5. Creates a CloudFront invalidation for /* to purge cached content from edge locations

Backend (CDK) deployments are performed manually. cdk deploy --all synthesises all eight stacks into CloudFormation templates and deploys them in dependency order.

10. Discussion: Design Decisions & Trade-offs

10.1 Presigned URL Upload vs. Proxy Through Lambda

Routing file bytes through Lambda is common in simpler architectures but imposes a hard 6 MB HTTP body limit and charges for Lambda execution time during the upload. The presigned URL pattern eliminates both constraints: S3 accepts files up to 5 GB, and Lambda only handles lightweight coordination messages. The cost is additional frontend complexity — two API calls per upload (presigned URL + confirm) instead of one.

10.2 Five API Gateways vs. One

A single API Gateway would simplify the frontend's URL management but would create a cross-stack resource dependency in CDK, preventing independent deployment of services. The five-gateway design mirrors the microservice boundaries: each stack fully owns its API surface with no shared infrastructure to coordinate. The frontend manages the five base URLs through Vite environment variables.

10.3 SNS Fan-out and the Notification Race Condition

The SNS fan-out pattern delivers analysis results to both the Results Worker and the Notification Lambda simultaneously — an elegant decoupling. However, the Notification Lambda needs the resultId (created by the Results Worker) to construct the report link. A retry loop (up to 15 s) compensates for this race condition. In production, including the resultId in the SNS message itself (computed before publishing) would eliminate the race entirely.

10.4 DynamoDB vs. Relational Database

DynamoDB was chosen for its zero-idle cost, serverless nature, and lack of VPC requirements. The trade-off is that every access pattern must be planned upfront through GSIs. This project has eight GSIs serving specific query shapes. Adding new ad-hoc queries post-deployment can be expensive (a new GSI must backfill from existing data). For complex analytics over user data, a relational store would be preferable.

10.5 Claude 3 Haiku vs. Higher-Capability Models

Claude 3 Haiku processes each resume prompt in approximately 5–8 seconds at a cost of roughly $0.006 per analysis. Claude 3.5 Sonnet would produce higher-quality output but at approximately 8× the cost and 2× the latency. For a student project at low volume, Haiku offers an excellent quality-to-cost ratio. A production system serving paying users would justify the upgrade.

10.6 Client-Side PDF Generation

Using @react-pdf/renderer in the browser eliminates a server-side PDF generation Lambda, reduces latency for the user, and avoids Lambda timeout concerns for large reports. The trade-off is a heavier JavaScript bundle (~1.7 MB uncompressed). Lazy-loading the PDF renderer component would mitigate initial load time in production.

10.7 Single-File Lambda Handlers

All service logic lives in one handler file per service (e.g., jobs_service/handler.py is approximately 660 lines). This simplifies deployment — no Lambda Layers or multi-file packaging — and makes each service self-contained. Production systems would decompose these into modules with shared utilities and dedicated unit tests per function.

10.8 What We Would Do Differently in Production

  • Restrict CORS to the specific CloudFront domain
  • Store API keys in AWS Secrets Manager rather than Lambda environment variables
  • Add CloudWatch Alarms on DLQ depth, Lambda error rates, and API 5xx rates
  • Enable AWS X-Ray tracing for end-to-end pipeline profiling
  • Add Provisioned Concurrency on API Lambdas to eliminate cold-start latency
  • Request SES production access to send to unverified addresses
  • Include resultId in the SNS message to remove the notification race condition
  • Add a comprehensive unit and integration test suite for each Lambda

11. Conclusion

Resume Analyzer demonstrates the power of composing managed cloud services into a coherent, production-quality platform without operating a single persistent server. The eight-microservice architecture — each independently deployed, independently scalable, and independently failure-isolated — illustrates the core principles of event-driven cloud design. The use of AWS CDK v2 Python for all infrastructure ensures the entire platform can be reproduced with a single command, a key property for maintainability and team collaboration.

The integration of Amazon Bedrock with real-time web search (Tavily) and live job-board data (Adzuna) demonstrates how LLM-powered features can be grounded in current, real-world context rather than static training knowledge. The post-analysis workflow — personalised job listings, AI-tailored resume generation, and a job-application tracker — transforms a one-time report into an ongoing career workflow tool.

The project resulted in a fully functional, publicly accessible web application deployed to AWS us-east-1, with an automated CI/CD pipeline for frontend deployments and a documented, reproducible backend deployment process. The architectural trade-offs identified in Section 10 provide a clear roadmap for productionising the system beyond the academic context.
